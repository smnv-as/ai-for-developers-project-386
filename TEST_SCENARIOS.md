# Тестовые сценарии Booking API

## Приоритеты

- **P0** — smoke tests, критический путь (E2E / интеграционные)
- **P1** — основные сценарии API
- **P2** — граничные случаи и бизнес-правила

## P0: Smoke Tests (E2E / интеграционные)

### UC-E1: Полный путь гостя
| Параметр | Значение |
|---|---|
| **Приоритет** | P0 |
| **Описание** | Гость проходит путь от главной страницы до успешного бронирования |
| **Шаги** | 1. Открыть `/` → отображаются типы событий <br> 2. Кликнуть «Выбрать» на любом типе → переход на `/slots?eventTypeId=` <br> 3. Выбрать любой доступный слот → переход на `/booking?eventTypeId=&startTime=` <br> 4. Заполнить форму: имя, email <br> 5. Нажать «Забронировать» |
| **Ожидаемый результат** | Успешное создание бронирования, редирект на `/success`, сообщение «Бронирование успешно!» |

### UC-E2: Админ создаёт тип события → гость видит его
| Параметр | Значение |
|---|---|
| **Приоритет** | P0 |
| **Описание** | После создания типа события админом гость сразу видит его на главной странице |
| **Шаги** | 1. Админ: `POST /admin/event-types` с `name="Новая консультация"`, `description="45 мин"` <br> 2. Гость: открыть `/` |
| **Ожидаемый результат** | На главной странице отображается карточка «Новая консультация» |

### UC-E3: Гость бронирует → админ видит бронирование
| Параметр | Значение |
|---|---|
| **Приоритет** | P0 |
| **Описание** | После успешного бронирования админ видит запись в панели бронирований |
| **Шаги** | 1. Гость: создать бронирование (UC-E1) <br> 2. Админ: открыть `/admin/bookings` |
| **Ожидаемый результат** | В таблице `/admin/bookings` присутствует запись с `guestName` и `guestEmail` из формы |

### UC-E4: Каскадное удаление типа события
| Параметр | Значение |
|---|---|
| **Приоритет** | P0 |
| **Описание** | При удалении типа события автоматически удаляются все связанные бронирования |
| **Шаги** | 1. Админ: создать тип события (UC-E2) <br> 2. Гость: создать бронирование этого типа (UC-E1) <br> 3. Админ: `DELETE /admin/event-types/{id}` |
| **Ожидаемый результат** | Тип события удалён. `GET /admin/event-types/{id}` → 404. `GET /admin/bookings` → удалённое бронирование отсутствует |

## P1: Публичный API

### UC-P1: Просмотр типов событий
| Параметр | Значение |
|---|---|
| **Приоритет** | P1 |
| **HTTP** | `GET /event-types` |
| **Шаги** | Отправить GET-запрос на `/event-types` |
| **Ожидаемый результат** | `200 OK`, массив `EventType[]` с полями `id`, `name`, `description` |

### UC-P2: Получение доступных слотов
| Параметр | Значение |
|---|---|
| **Приоритет** | P1 |
| **HTTP** | `GET /slots?eventTypeId={id}&from={date}&to={date}` |
| **Шаги** | 1. Выбрать существующий `eventTypeId` <br> 2. `GET /slots?eventTypeId=type-1&from=2026-06-08&to=2026-06-12` |
| **Ожидаемый результат** | `200 OK`, массив `Slot[]` с полями `startTime`, `endTime`. Каждый слот длится ровно 30 мин, только пн–пт, только 09:00–17:30, только в пределах 14-дневного окна, только на :00 или :30 |

### UC-P3: Успешное бронирование
| Параметр | Значение |
|---|---|
| **Приоритет** | P1 |
| **HTTP** | `POST /bookings` |
| **Тело** | `{ "eventTypeId": "...", "startTime": "2026-06-09T10:00:00Z", "guestName": "Иван", "guestEmail": "ivan@example.com" }` |
| **Шаги** | Отправить POST с валидными данными |
| **Ожидаемый результат** | `200 OK`, объект `Booking` с заполненными `id`, `startTime`, `endTime`, `eventTypeId`, `guestName`, `guestEmail` |

### UC-P4: Отмена бронирования
| Параметр | Значение |
|---|---|
| **Приоритет** | P1 |
| **HTTP** | `DELETE /bookings/{id}` |
| **Шаги** | 1. Создать бронирование (UC-P3) <br> 2. `DELETE /bookings/{bookingId}` |
| **Ожидаемый результат** | `204 No Content`. Последующий `GET /admin/bookings` не возвращает эту запись (hard delete) |

### UC-P5: Ошибка — слот уже занят
| Параметр | Значение |
|---|---|
| **Приоритет** | P1 |
| **HTTP** | `POST /bookings` |

## P2: Бизнес-правила и граничные случаи

### UC-B1: Длительность слотов = 30 минут
| Параметр | Значение |
|---|---|
| **Приоритет** | P2 |
| **Описание** | Каждый слот длится ровно 30 минут |
| **Шаги** | `GET /slots?eventTypeId=` → проверить все слоты |
| **Ожидаемый результат** | Для каждого слота `slot.endTime - slot.startTime == 30 минут` |

### UC-B2: Слоты только по будням
| Параметр | Значение |
|---|---|
| **Приоритет** | P2 |
| **Описание** | Слоты генерируются только на пн–пт |
| **Шаги** | `GET /slots?eventTypeId=` → проверить даты всех слотов |
| **Ожидаемый результат** | Ни один слот не приходится на субботу или воскресенье |

### UC-B3: Слоты только в рабочие часы
| Параметр | Значение |
|---|---|
| **Приоритет** | P2 |
| **Описание** | Слоты генерируются только в диапазоне 09:00–18:00 Europe/Moscow |
| **Шаги** | `GET /slots?eventTypeId=` → проверить время всех слотов |
| **Ожидаемый результат** | Все слоты начинаются в диапазоне 09:00–17:30 (последний слот начинается в 17:30, заканчивается в 18:00). Нет слотов начинающихся до 09:00 или в 18:00+. |

### UC-B4: Границы слотов :00 и :30
| Параметр | Значение |
|---|---|
| **Приоритет** | P2 |
| **Описание** | Начало каждого слота выровнено на xx:00 или xx:30 |
| **Шаги** | Проверить `startTime` каждого слота |
| **Ожидаемый результат** | Минуты `startTime` равны 0 или 30 |

### UC-B5: Окно бронирования — 14 дней
| Параметр | Значение |
|---|---|
| **Приоритет** | P2 |
| **Описание** | Доступны только слоты в пределах 14 дней от текущей даты |
| **Шаги** | Проверить, что все слоты не дальше 14 дней от today |
| **Ожидаемый результат** | Все слоты находятся в диапазоне `[today, today + 14 days]` |

### UC-B6: Hard delete при отмене
| Параметр | Значение |
|---|---|
| **Приоритет** | P2 |
| **Описание** | Отмена бронирования полностью удаляет запись из БД |
| **Шаги** | 1. Создать бронирование (UC-P3) <br> 2. Отменить его (UC-P4) <br> 3. Попытка получить информацию о бронировании |
| **Ожидаемый результат** | Запись полностью отсутствует в системе. Нет статуса «отменено» — только физическое удаление. |

## Известные расхождения

### Каскадное удаление EventType → Bookings

| Компонент | Текущее поведение | Ожидаемое поведение |
|---|---|---|
| **TypeSpec** (`main.tsp`) | «deleting an event type cascades to its bookings» | Каскадное удаление |
| **Backend** (`service.go:DeleteEventType`) | Возвращает `EventTypeHasBookingsError` если есть связанные бронирования | Должен удалять тип события **и все его бронирования** |
| **Frontend** (Admin UI) | React-admin покажет ошибку при попытке удаления | Удаление проходит успешно, связанные бронирования также удаляются |

**Статус:** требует исправления в `backend/internal/service/service.go` и `backend/internal/repository/repository.go`.

## Нотация

- `{id}` — подставить реальный идентификатор из предшествующего шага
- `{date}` — дата в формате `YYYY-MM-DD`
- `{utcDateTime}` — дата и время в формате ISO 8601 (RFC3339), например `2026-06-09T10:00:00Z`
- Все времена в UTC, если не указано иное. Сервис преобразует к Europe/Moscow для применения бизнес-правил.
